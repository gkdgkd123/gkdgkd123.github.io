我的第一篇文章

这篇是对 Java 反序列化问题的一个简要梳理，偏实践视角，尽量把核心概念、典型风险点和防护要点讲清楚。

1. 反序列化到底“危险”在什么地方？

Java 原生序列化会把对象图完整写入字节流，反序列化时再完整还原。问题在于：反序列化并不只是“还原数据”，而是会触发对象的构造、字段还原与回调方法（比如 readObject()、readResolve()、finalize() 等）。

如果输入的字节流来自不可信来源，就可能被精心构造，借助应用依赖中的“可利用类”（gadget）拼出一条执行链，最终达到任意方法调用甚至远程代码执行。

2. “Gadget Chain” 是怎么形成的

很多第三方库里存在“看起来正常但组合起来危险”的类，比如某些集合、反射相关、模板引擎、JNDI 组件等。攻击者会把这些类按特定结构序列化，反序列化时触发一连串方法调用，这就是所谓 gadget chain。

社区里常用的测试工具是 ysoserial，它集合了一批常见库的利用链，很多真实漏洞也都是这些思路的变体。

3. 常见触发场景

  • 使用原生 ObjectInputStream 反序列化来自网络/文件/缓存的对象
  • RPC 或消息系统使用 Java 序列化作为默认协议
  • 历史系统中把序列化对象直接存库或放到 Redis
  • Web 容器/中间件存在历史序列化接口（JMX、RMI、JNDI 等）

4. 防护思路（优先级从高到低）

  1. 完全避免反序列化不可信数据（最重要）
    • 能换 JSON/Protobuf 就别用 Java 原生序列化
  2. 启用类白名单/黑名单
    • JDK 9+ 提供 ObjectInputFilter
  3. 升级依赖库
    • 很多 gadget 来自旧版本库
  4. 隔离与最小化依赖
    • 依赖越少，可利用面越小
  5. 安全监控
    • 反序列化异常、类加载异常值得重点关注

5. 一个简单的排查思路

  • 先在代码里搜索 ObjectInputStream / readObject / deserialize
  • 看入参是否来源于外部输入
  • 列出当前依赖库版本，对照历史可利用链
